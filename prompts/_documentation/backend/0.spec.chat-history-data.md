# Chat History — Data Shape

DB'de chat history için ne tutuyoruz. Java tarafı kapsam dışı (`6.task.agent-chat-persistence.md`'de).

## Tablo

```sql
CREATE TABLE agent_chat_message (
    id            BIGSERIAL PRIMARY KEY,
    workspace_id  BIGINT NOT NULL,
    project_id    BIGINT NOT NULL,
    user_id       BIGINT NOT NULL,
    turn_index    INTEGER NOT NULL,           -- conversation içindeki sıra (1, 2, 3, ...)
    role          VARCHAR(20) NOT NULL,       -- 'user' | 'assistant' | 'system'

    -- Display alanları (her satırda dolu olabilir)
    content       TEXT,                       -- ana metin (chat_response.text veya user input)
    hint          TEXT,                       -- ipucu (chat_response.hint)
    next_actions  VARCHAR(120),               -- '[NA:GENERATE|...]' tag

    -- Routing metadata (sadece assistant satırlarında)
    action        VARCHAR(40),                -- 'structure_generated' | 'enrich_questions' | ...
    intent        VARCHAR(20),                -- 'builder' | 'analyze' | 'interview' | ...
    variant_key   VARCHAR(40),                -- 'add_features' | 'enrich_context' | ...
    agent_chain   TEXT[],                     -- ['INTENT_ROUTER', 'BUILDER']
    action_hint_override VARCHAR(60),         -- direct route ise dolu

    -- Card payload (sadece interactive card üreten action'larda)
    card_data     JSONB,                      -- card snapshot (per-action shape — aşağıda)
    card_status   VARCHAR(20),                -- 'pending' | 'inserted' | 'discarded' | 'merged' | 'active' | 'completed'

    -- Debug / replay
    request_payload   JSONB,                  -- backend → LLM (her LLM call ayrı entry'de)
    response_payload  JSONB,                  -- LLM → backend

    status        VARCHAR(20) DEFAULT 'ACTIVE',
    created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
    created_by    BIGINT,

    CONSTRAINT fk_agent_chat_workspace FOREIGN KEY (workspace_id) REFERENCES workspace(id),
    CONSTRAINT fk_agent_chat_project   FOREIGN KEY (project_id)   REFERENCES project(id)
);

CREATE INDEX idx_agent_chat_project_turn ON agent_chat_message (project_id, turn_index);
CREATE INDEX idx_agent_chat_user         ON agent_chat_message (user_id);
```

## Common alanlar — her satır

| Kolon | Açıklama |
|---|---|
| `turn_index` | Project içinde sıralı integer. Aynı turn'de birden fazla LLM call tek satırda — `request_payload` array. |
| `role` | `user` (kullanıcı yazdı) / `assistant` (LLM cevabı) / `system` (otomatik mesaj, ör. "🔄 Generating...") |
| `content` | Görünür metin. User'da raw input, assistant'ta `chat_response.text`. |
| `hint` | `chat_response.hint` — opsiyonel ipucu. |
| `next_actions` | `[NA:...]` tag pass-through. Frontend NA bar'ı buradan render. |
| `action` | Card türü ayırıcı. Yoksa text-only mesaj. |
| `card_data` | Card render için snapshot (action'a göre shape). |
| `card_status` | Card'ın yaşam döngüsü. Text mesajda null. |

## Action türleri ve `card_data` shape'leri

### `structure_generated` — BUILDER tree

**Üreten mode'lar:** `generate_modules`, `generate_modules_features`, `add_features`, `add_subfeatures`

**`card_data`:**
```json
{
  "sections": [
    {
      "id": "string",
      "name": "string",
      "modules": [
        {
          "id": "string",
          "name": "string",
          "summary": [{ "text": "string", "source": "string | null" }],
          "assumptions": ["string"],
          "color": "string",
          "features": [
            {
              "id": "string",
              "name": "string",
              "summary": [{ "text": "string", "source": "string | null" }],
              "assumptions": ["string"]
            }
          ]
        }
      ]
    }
  ],
  "generation_type": "modules | modules_features | features | subfeatures",
  "is_diff": false
}
```

**`card_status`:**
- `pending` — kullanıcı henüz Insert/Discard basmadı
- `inserted` — Insert edildi, DB'ye yazıldı
- `discarded` — kullanıcı Discard etti

### `enrich_questions` — INTERVIEW enrich start

**Üreten mode:** `enrich_context` (status=start)

**`card_data`:**
```json
{
  "questions": [
    {
      "id": "string",
      "text": "string",
      "type": "text | single_choice",
      "question_index": 0,
      "total_questions": 0,
      "options": [
        { "id": "string", "label": "string", "command": "string" }
      ]
    }
  ],
  "depth": "quick | medium | deep"
}
```

**`card_status`:**
- `active` — Q&A devam ediyor (kullanıcı henüz Submit basmadı)

**Refresh davranışı:** B yaklaşımı — soru snapshot'ı görünür, verilen cevaplar sıfırlanır (cevaplar persist edilmez), user tekrar cevaplar.

### Card'sız action'lar

Aşağıdakilerde `action` dolu ama `card_data` ve `card_status` **null**:

| `action` | Açıklama |
|---|---|
| `builder_inserted` | Insert sonrası confirmation mesajı |
| `builder_discarded` | Discard confirmation |
| `mutation_done` | MUTATION sonucu |
| `clarify` | CLARIFY agent cevabı |
| `enrich_completed` | Enrich submit sonrası "Context enriched" mesajı |
| `solve_completed` | Solve sonrası "Resolved X assumptions" mesajı |
| `answer` | ANSWER agent normal cevap |
| `analyze_input` (chat metni) | ContextCard ayrı satırda; bu satır chat metni only |

Bunlar `content` + `next_actions` ile yeterli — UI text bubble basar.

## Card status state machine

```
                  ┌──── inserted/merged
                  │
     pending  ────┤
                  │
                  └──── discarded


     active  ────────── completed   (interview)


     null (no status)                (context_card, text-only messages)
```

Status transition'ları başka turn'lerde olur:
- `pending → inserted`: kullanıcı "Insert" → yeni turn yaratılır + önceki card mesajının status'ü update edilir
- `pending → discarded`: kullanıcı "Discard" → aynı pattern
- `active → completed`: enrich/solve Submit → aynı pattern

Yani **bir turn iki şey yapabilir:**
1. Yeni satır INSERT (yeni mesaj)
2. Önceki bir satır UPDATE (`card_status` değişir)

Frontend update'i fark edebilmek için restore endpoint güncel statuses'ları döner.

## Restore endpoint — neyi döner

```sql
SELECT
    turn_index, role, content, hint, next_actions,
    action, card_data, card_status,
    intent, variant_key, agent_chain, action_hint_override,
    created_at AS timestamp
FROM agent_chat_message
WHERE project_id = :projectId
  AND status = 'ACTIVE'
ORDER BY turn_index ASC
LIMIT :limit
```

Frontend bu satırları olduğu gibi alır:
- Text mesajları → bubble render
- Card'lı satırları → `card_data` + `card_status`'e göre component render
- Latest pending → aktif card (Insert/Discard butonlu)
- Inserted/discarded/completed → read-only badge'li görünüm

Debug payload'ları (`request_payload`, `response_payload`) restore'da **dönmez** — sadece admin/debug ekranı isterse ayrı endpoint çekilir.

## Özet — ne tutuyoruz

| Veri | Nerede | Notes |
|---|---|---|
| Sıra | `turn_index` | her satırda |
| Aktör | `role` | user/assistant/system |
| Görünür metin | `content`, `hint` | her assistant satırında |
| NA tag | `next_actions` | NA bar için |
| Card türü | `action` | render kararı |
| Card içeriği | `card_data` jsonb | per-action shape |
| Card yaşam döngüsü | `card_status` | pending/inserted/active/completed/... |
| Routing trace | `intent`, `variant_key`, `agent_chain`, `action_hint_override` | observability |
| LLM I/O | `request_payload`, `response_payload` jsonb | debug only |

**Ne tutmuyoruz (intentional):**
- Interview cevap progress'i (mid-flow answer state) — refresh'te kaybolur (B yaklaşımı)
- Frontend transient state (typing, input field içeriği)
- Agent system prompt (DB'de zaten ayrı tablo)
