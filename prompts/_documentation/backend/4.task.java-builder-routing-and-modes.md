# LLM I/O Contract Update — Java Backend Task

## Sorun

Prompt'lar ve frontend güncellendi; Java backend henüz yeni contract'ı
tanımıyor. Yeni alanlar/modlar request'te gelince Java ya null atıyor
ya da "unknown mode" hatası veriyor.

**Referans contract:** [0.spec.llm-io-contract.md](0.spec.llm-io-contract.md)

## Değişiklikler (Özet)

| # | Konu | Durum |
|---|---|---|
| 1 | `action_hint_override` — INTENT_ROUTER bypass | Frontend yolluyor, Java okumuyor |
| 2 | `targets[]` — minimal context payload | Frontend yolluyor, Java forward etmiyor |
| 3 | `add_features` / `add_subfeatures` BUILDER modes | Prompt tarafı hazır, Java enum'da yok |
| 4 | `variant_key` terminolojisi | Prompt'larda `action_hint` → `variant_key` |
| 5 | `variant_key_null` state | `action_hint_null` → `variant_key_null` rename |
| 6 | Free-text name resolution | "Generate features for Auth" → targets[] parse |
| 7 | Synthetic ID → DB UUID mapping | `mod_*`, `feat_*`, `subfeat_*` client-side resolve |

## 1. `action_hint_override` — Direct Route

### Request shape

```json
{
  "user_input": "Generate Features for current modules",
  "action_hint_override": "BUILDER:add_features",
  "targets": [...],
  "workspace_id": "...",
  "project_id": "..."
}
```

### Davranış

`action_hint_override` set ise:
- INTENT_ROUTER **çağrılmaz**
- Doğrudan hedef agent + variant dispatch edilir
- `agentChain` sadece hedef agent'ı içerir (örn. `[BUILDER]`)

### Fix

`AgentChatRequest.java` DTO'ya alan ekle:

```java
@JsonProperty("action_hint_override")
private String actionHintOverride;   // "BUILDER:add_features" formatında

@JsonProperty("targets")
private List<Map<String, Object>> targets;
```

`AgentOrchestrator.chat()` başında:

```java
if (request.getActionHintOverride() != null) {
    var parts = request.getActionHintOverride().split(":");
    String agent = parts[0];           // BUILDER
    String variantKey = parts[1];      // add_features
    return dispatchDirect(agent, variantKey, request);
}
// yoksa normal akış: INTENT_ROUTER → dispatch
```

## 2. `targets[]` Forward

### Neden

`add_features` / `add_subfeatures` mode'ları LLM'e **sadece hedef element'i**
yolluyor — sibling tree değil. Java bu alanı prompt input'una pass-through
yapmalı.

### Shape (add_features)

```json
"targets": [
  {
    "id": "<module UUID>",
    "name": "Auth",
    "summary": [...],
    "assumptions": [...],
    "color": "#...",
    "existing_features": ["Login", "Signup"]
  }
]
```

### Shape (add_subfeatures)

```json
"targets": [
  {
    "id": "<feature UUID>",
    "name": "Login",
    "summary": [...],
    "assumptions": [...],
    "existing_subfeatures": ["Email Login"],
    "parent_module": {
      "id": "...", "name": "...", "summary": [...], "assumptions": [...]
    }
  }
]
```

### Fix

`BuilderAgent.buildInput()` içinde:

```java
if (request.getTargets() != null) {
    promptInput.put("targets", request.getTargets());
}
```

## 3. `add_features` / `add_subfeatures` BUILDER Modes

### Yeni enum değerleri

`BuilderMode` enum'a (veya mode string listesine) ekle:

```java
public enum BuilderMode {
    GENERATE_MODULES("generate_modules"),
    GENERATE_MODULES_FEATURES("generate_modules_features"),
    GENERATE_PAGES("generate_pages"),
    ADD_FEATURES("add_features"),          // YENİ
    ADD_SUBFEATURES("add_subfeatures"),    // YENİ
    DIFF("diff"),
    RESOLVE("resolve");
}
```

### DB kayıtları (hazır)

- `ai_agent_prompt` tablosunda `BUILDER_MODE_ADD_FEATURES` (id: 35)
- `ai_agent_prompt` tablosunda `BUILDER_MODE_ADD_SUBFEATURES` (id: 36)
- `conditional_on: {"mode": ["add_features"]}` / `{"mode": ["add_subfeatures"]}`

Java `AgentPromptComposer`'ın `mode` eşleştirmesi zaten generic — yeni
mode string'leri destekler.

### Output shape

**add_features response:**
```json
{
  "modules": [
    {
      "id": "<target module UUID>",
      "features": [
        { "id": "feat_<synthetic>", "name": "...", "summary": [...], "assumptions": [] }
      ]
    }
  ]
}
```

**add_subfeatures response:**
```json
{
  "features": [
    {
      "id": "<target feature UUID>",
      "subfeatures": [
        { "id": "subfeat_<synthetic>", "name": "...", "summary": "..." }
      ]
    }
  ]
}
```

### Fix

Response handler'da yeni mod'ları parse et. Frontend `modules[]` /
`features[]` array'lerini iterate edip synthetic ID'leri real UUID'ye map'liyor
(bkz. #7). Java sadece pass-through yapsın.

## 4. `variant_key` Terminoloji

Prompt'larda `action_hint` → `variant_key` rename edildi. Java tarafı
internal olarak hâlâ `actionHint` diyor olabilir — isim uyumu kritik değil
ama log ve error mesajlarında kafa karışıklığı yaratır.

### Opsiyonel rename

- `CompositionRequest.actionHint` → `variantKey`
- Log'larda `"actionHint={}"` → `"variantKey={}"`

Şart değil, prompt-facing bir değişiklik değil. Sadece `action_hint_override`
field ismi değişmedi (frontend yolluyor) — o kalsın.

## 5. `action_hint_null` → `variant_key_null` State Key

### Mevcut

`AgentPromptComposer` conditional rule'lar için state flag üretiyor:

```java
Set<String> state = new HashSet<>();
if (req.actionHint() == null) state.add("action_hint_null");
```

### Prompt tarafı

`ANALYZE_RULE_FALLBACK_DETECTION.md` artık `{"state": ["variant_key_null"]}`
conditional'ı kullanıyor. `action_hint_null` bulamazsa fallback rule
tetiklenmez.

### Fix

```java
if (req.variantKey() == null) state.add("variant_key_null");
```

(Veya hem eskisini hem yenisini bir süre yollayabilirsin — geçiş için.)

## 6. Free-text Name Resolution

### Senaryo

User chat'e yazar:

> "Generate features for the Auth module"

INTENT_ROUTER → `builder:add_features`. Ama `targets[]` yok (frontend
göndermedi). Java'nın kendisi modül adını parse edip targets[] kurması gerek.

### Fix

`BuilderAgent.handle()` içinde `add_features` mode için:

```java
if ("add_features".equals(variantKey) && request.getTargets() == null) {
    List<String> moduleNames = extractModuleNames(request.getUserInput());
    // "all" veya isim yok → tüm modüller
    if (moduleNames.isEmpty() || moduleNames.contains("all")) {
        targets = loadAllModules(projectId);
    } else {
        targets = resolveModulesByName(projectId, moduleNames);
    }
    promptInput.put("targets", targets);
}
```

`add_subfeatures` için aynı mantık, feature seviyesinde + parent_module dolur.

### Ambiguity

İsim birden fazla modülle eşleşirse (farklı section'larda) → `clarify` action
ile kullanıcıya sor. MUTATION_AGENT'ta aynı pattern var, oradan örnek al.

### İsim Çıkarma

Basit regex veya küçük bir LLM sub-call. Önerim: regex ile başla
("for the X module", "to Y module"), yetmezse LLM'e sor.

## 7. Synthetic ID → DB UUID Mapping

### Durum

LLM output'unda yeni elementlerin ID'leri `mod_xyz`, `feat_xyz`, `subfeat_xyz`
prefix'li **synthetic** string'ler. DB insert sırasında real UUID
üretilmeli.

### Mapping Kimde?

**Client-side** (frontend) yapıyor. `agent-chat.store.ts` response'u alıp
yeni elementler için UUID üretiyor ve backend'e create isteği atıyor.

Java tarafı synthetic ID'leri **görmezden gelebilir** — response'u
frontend'e pass-through yapsın. Ama eğer server-side insert eklenirse
(ilerleyen faz), aynı mapping Java'da da gerekli olur.

### Kural

Prefix'li ID = yeni element (DB'de yok). Prefix'siz UUID = mevcut element
(target ID, preserve edilmeli).

```java
boolean isSynthetic(String id) {
    return id.startsWith("mod_") || id.startsWith("feat_") || id.startsWith("subfeat_");
}
```

## Test

**T1 — Direct route:**
- Given: `actionHintOverride = "BUILDER:add_features"`, `targets = [...]`
- Then: `agentChain == [BUILDER]`, INTENT_ROUTER çağrılmaz

**T2 — Free-text name resolution:**
- Given: `user_input = "Generate features for Auth"`, `targets = null`
- When: INTENT_ROUTER → `builder:add_features`
- Then: Java "Auth" modülünü bulup targets[] kurar, prompt'a yollar

**T3 — Ambiguity:**
- Given: iki section'da "Auth" modülü var
- Then: `action: "clarify"`, hangi section sorusu

**T4 — existing_features[] duplicate prevention:**
- Given: Auth modülünde "Login" zaten var
- When: `add_features` çalıştı
- Then: LLM "Login" adını tekrar üretmez (prompt'a `existing_features` yollanmış)

**T5 — state flag rename:**
- Given: request'te `variantKey == null`
- Then: composition state `variant_key_null` içerir
