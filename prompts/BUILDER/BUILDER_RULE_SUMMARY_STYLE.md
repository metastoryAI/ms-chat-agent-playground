## SUMMARY STYLE

### Sentence count per summary
- **Default — both modules and features: 1–2 sentences.** Features almost always stay at 1–2.
- **Module exception — 3rd sentence ONLY** when there is a real integration or external dependency worth naming (e.g. "Integrates with the project's payment provider for capture."). If the third sentence would just elaborate or restate the first two, drop it.

### Structure (when using multiple sentences)
- Sentence 1: The core action — what does this do? One main verb.
- Sentence 2: The mechanism — how does it work, or what state/structure does it use?
- Sentence 3 (modules only, only when warranted): Integration or external dependency.

Use fewer sentences when the item is simple — 1 sentence is fine when it captures the whole story.

### Sentence length — short and human-readable
- **Each sentence is one self-contained idea ending with a period.** Never run two ideas together into one sentence.
- **Aim for 10–20 words per sentence; never exceed ~25.** If a sentence grows past that, it's a sign you're chaining ideas — split it.
- A sentence that needs reading twice to parse is too long. Rewrite for clarity, not for elegance.
- No comma-chains of 3+ items. No participle-clauses ("Verarbeitung X, was wiederum…"). No long subordinate clauses ("welches dafür sorgt, dass…").

### Splitting rule
If you find yourself writing "and", "as well as", or a comma-list of 3+ items inside one sentence, split into the next role instead. Every "and" joining two independent ideas becomes a period.

### Voice and word choice
Active voice. Concrete verbs. Avoid filler ("enables", "facilitates", "is responsible for", "serves to").
